## **3) 발생할 수 있는 상황**

<br>

### Q1. git commit 까지만 쓰면?

- git commit 까지만 쓰고 나서 편집기에서 메시지를 좀 더 길게 쓸 수 있음

- 이후 git log --oneline 으로도 확인 가능

<br>

### Q2. 원격저장소 이름이나 유저명을 바꾼다면?

- 원격저장소 이름이나 GitHub username을 바꾼다면 원격 저장소 설정을 해야 함 (get remote 명령어 주소)

- git remote add origin 주소

- `깃아` / `원격저장소` / `추가해줘` /  `오리진으로` / `주소를`

<br>

### Q3. 프로젝트 진행 시 공유해서는 안 되는 내용이 있다면?

- 개인 프로젝트에 필요하지만 공유되면 안되는 것들
    
    - 개발 관경과 관련된 파일들
    
    - 운영체재 / 개발언어 / IDE

    - 환경 변수 등

- .gitignore로 해두면 안나오게 할 수 있음 (커밋하고 싶을 경우 add 해야함)

- https://www.toptal.com/developers/gitignore/

- 위 사이트에서 자주 쓰는 언어들을 입력하면 전세계 개발자들의 gitignore 설정을 확인 가능

- *.csv 확장자 앞에 *을 붙이면 안나옴

-  이미 커밋된 파일은 반드시 삭제를 해야 .gitignore로 적용 (프로젝트 시작 전 미리 설정)

<br>

### Q4. GitHub 페이지에서 바로 바꾸면 로컬에서는?

## **4) Branch & Merge**

### **필요한 이유**

- 프로젝트 시 독립적인 작업 흐름을 만들고 관리

- 각자 작업한 파일들을 취합하는 것이 어려움

- 협업을 하다 보면 엄청나게 많은 소스 코드들이 존재

- 각 브랜치에서 작업을 한 이후 합치기 위해 merge 사용

```
**예시

카카오 메시지 플랫폼
- 배경 10.6.01
- 댓글 기능 10.6.02
- 송금 10.6.01

사용자들 사용 : 카카오톡 버전 10.5.3 (6/8 기준)

10.6 
10.6.01
10.6.02

개발자들은 개발하면서 이후 버전을 쌓고 있음
```

- 브랜치 개별 개념 (구분은 회사마다 다 다름)

    - `master(main)` : 제품으로 출시될 수 있는 브랜치

    - `develop` : 다음 출시 버전을 개발하는 브랜치

    - `feature` : 기능을 개발하는 브랜치

    - `release` : 이번 출시 버전을 준비하는 브랜치

- 서로 다른 작업물이 겹쳤을 때 합치는 사람이 둘 중 하나를 최종 선택해서 합칠 수 있음

- pull : 충돌되는 것들을 수정할 수 있도록 가져오는 것

- 브랜치를 나눠서 작업하면 독립된 작업 공간을 반영해서 업데이트를 할 수 있음

- 브랜치는 독립된 작업의 흐름을 보장해준다 

- 왜냐하면 합치는 순간(merge)에만 반영을 해주니까

- merge : commit이 자동으로 발생

<br>

### **branch 명령어**

- 생성 : git branch [branch_name]

- 이동 : git checkout [branch_name]

- 새로운 브랜치 생성 및 전환 : git checkout -b [branch_name]

- 브랜치 목록 보기 : git branch

- 브랜치 삭제하기 : git branch -d [branch_name]

- 원격 브랜치 가져오기 : git fetch [remote_name]

- 원격 브랜치로 전환하기 : git checkout [remote_name]/[branch_name]

## **5) PULL**

- git pull <원격저장소이름> <브랜치 이름>

- 원격저장소 프로젝트 시작 (원격저장소 복제 후 가져옴) 

- git clone 저장소 주소

    - git clone https://github.com/heejindays/git-0608
    
- clone : 원격저장소 복제

- pull : 원격저장소 커밋 가져오기

<br>

### **기본 개념**

- 로컬에서 새로운 프로젝트 시작 : git init

- 원격에 있는 프로젝트 시작 : get clone

- 프로젝트 개발 중 다른 사람 커밋 받아오기 : git pull

- 내가 한 로컬 프로젝트 개발 공유 : git push

- pull request : 관리자에게 내가 지금 개발한 내용들을 반영해서 merge 해달라고 요청 (내 저장소에서 요청)

<br>

### **push 실패할 경우**

- 원격저장소의 커밋을 원격저장소로 가져와서 (pull)

- 로컬에서 두 커밋을 병합 (추가 커밋 발생)

- 다시 GitHub으로 push

## **6) FORK로 협업**

<br>

### **작업 순서**

**1. 원본 저장소 포크**

- 원하는 GitHub 페이지에서 원본 저장소에 접속

- 우측 상단에 위치한 `Fork` 버튼을 클릭

- GitHub 계정을 선택하고 포크할 저장소의 이름을 확인


<br>

**2. 포크한 저장소로 작업**

- 포크한 저장소로 이동

- 자신의 계정 아래에 있는 저장소로 작업

- 코드를 수정하거나 새로운 기능을 추가

- 커밋을 하여 변경 사항을 저장

<br>

**3. 원본 저장소와 동기화**

- 변경 사항이 있을 경우, 포크한 저장소와 동기화

- 자신의 포크한 저장소 페이지로 이동

- 우측 상단에 있는 `Pull requests` 버튼을 클릭

<br>

**4. 변경 사항 제출하기**

- 포크한 저장소에서 작업한 내용을 

- 원본 저장소에 반영하고자 할 경우 

- `Pull request`를 생성하여 변경 사항을 제출

- `Pull requests` 탭으로 이동

- `New pull request` 버튼 클릭

- 변경 사항을 확인하고 설명을 추가

- `Create pull request` 버튼을 클릭

- 원본 저장소 소유자에게 변경 사항을 알림
